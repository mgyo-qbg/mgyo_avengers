## 이슈 라이프사이클

```
Phase 1. 요구사항 수집
    사용자가 요구사항을 설명 (부분적이어도 됨)
    Claude가 궁금한 사항을 질문 → 사용자가 답변
    이 대화를 반복하며 spec.md 초안 작성

Phase 2. 스펙 초안
    spec.md 초안 완성 → 사용자에게 검토 요청
    ↓
    [기획자 게이트 검토]
    - 앱 이슈: 앱-기획자가 app_planner_context.md 참조하여 검증
    - 웹 이슈: 웹-기획자1이 web_planner_context.md 참조하여 검증
    검증 항목:
      · 기존 구현 화면/기능과 충돌 여부
      · 진행 중 이슈와 요건 상충 여부
      · 확정된 UX/아키텍처 원칙 위반 여부
    이슈 발견 시 → 사용자에게 보고 후 spec.md 수정

Phase 3. 스펙 확정  ← 이 단계 이전에는 절대 작업 착수하지 않는다
    사용자 최종 승인
    spec.md 상태 → "스펙확정"으로 변경
    MEMORY.md "현재 진행 중 이슈" 업데이트
    기획자 컨텍스트의 "진행 중 이슈" 업데이트

Phase 4. 작업 진행
    담당팀에 작업 할당
    구현 완료 후 results.md 작성
    필요 시 사용자 테스트 진행 → user_test_results.xlsx 기록

Phase 5. 완료
    사용자에게 결과 보고
    기획자 컨텍스트의 "제품 현황" 업데이트 (새로 구현된 기능 반영)
    기획자 컨텍스트의 "확정된 원칙" 업데이트 (이슈에서 도출된 원칙 추가)
    MEMORY.md "현재 진행 중 이슈" → 완료 처리
```
